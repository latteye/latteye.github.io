How's it going?

2009年9月26日

细细看看其实今年就没认真的写过 Blog，或者说，兴趣上的事情就没认真过。

年初的时候写 BearAudio，写了两个月，搁浅。
不再更新技术 Blog，做了好多事情，都没整理。
Linux 桌面也不再玩了，虽然更新了一次新的硬件。
前两天开了一个 SA 服务器，简直就是悲剧。有些时候我自以为我适合做服务性行业，但真做了，我总是向往着做纯技术的无忧无虑。
和成批的人接触简直就是悲剧，而更因为网络的虚拟化，我所接触的简直就不是人，而是各个人背后的鬼影，一切卑劣的品质暴露无遗。
当然，不排除虚拟的环境中依旧有好人，笑看风云是一个。

就当是看了一部“七宗罪”。

卍

花了三、四天时间重新整理了一下 Blog 模板。
加入了 GG 广告。
重新调整了繁琐的分类。
请公司的小妹妹帮忙做了 Logo。

跑来跑去的找合适的主机，却因为域名通不过备案，今天又迁移回了 imtx 的 HostMonster 。话说 HostMonster 大多数时候还是不错的。可是前两天主机上总有人在占用大量 CPU，我猜测是在做批量 html 生成。Load Average 一度飙至 20-40 。大多数用户的 php 程序一概无响应。结果 HostMonster 不闻不问！
在 TX 同学好心提交问题之后，HostMonster 居然创新的提出，主机速度慢是由于我的网站文件数量太多而导致。（当时我有个网站 html 数量达到了 50 万个……）同时，HostMonster 要求我删除文件，不然就禁用 TX 的账号。在删除文件的同时，我向 TX 表达了对 HostMonster 的愤慨，呵呵。

不过我后来和老板一起讨论了这个问题，究竟文件数量达到这个量级之后，对系统的影响有多大？
1. inode 的占用不用多说了，对磁盘的浪费的确很大。对这个概念不熟悉的同学可以看看 鸟哥的文章。
2. 老板认为如此多的文件数量，apache 在做“一个操作”（抱歉，忘记老板那天是怎么说的了）的时候，会有很大影响。
3. 老板说 buffer 的增加会很厉害，不熟悉 buffer 概念的同学可以看看 IBM 的一篇文章。

后来我还很有兴趣的询问了 IX 这家虚拟主机商，他们能提供多少文件。得到的答复是 10 万个文件。
国外大多数虚拟主机都是不限制空间的（Storage），但是隐含的限制文件数量。所谓隐含，是他们在协议中不会说明。以前在使用 DreamHost 的时候也用过几十万的文件。可能是 DH 没发现吧，不然也会警告我。

卍

工作上倒是有很多进展。

年初的时候为公司机房做搬迁准备，在 LVS、网卡、Oracle 方面做了很多工作。随后的整理也花了不少时间 Nagios ，Cacti。但总觉得还不够，还有些问题没有解决。
其中 Cacti 存在无规律（至少还没找到规律）掉数据的情况。
Oracle 存在 ASM 无 hugepage 死掉的情况，也还没解决。

而这两天主要在配合其他小组的人做一套 3g 网卡自动拨号+高级路由的东西。系统是 Ubuntu，通过 ppp 的各项功能做到自动拨号并为每个 pppx 链接创建合适的路由表。大概也做了一个礼拜了。
虽然东西能用了，但有很多知识点也没有好好深入，udev 等等也是一知半解。

兴趣上有两个事情要解决。

一是 BearAudio。
二是 想做一套 mail-list-archive 的系统。看到了几个工具，都是生成 html 的，但是虚拟主机商提出了文件数量的限制，限制要想办法把东西放在数据库里面了。

Linux 桌面不玩了，真的不玩了……自己用着 debian ，基本上都用 stable。系统么，够用就好，能工作就好了。

写了这些，算回顾也算开篇。
